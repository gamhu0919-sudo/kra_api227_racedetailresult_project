# API15_2 JSON 구조 전처리 규칙 기준안 (Preprocessing Rules)

기존 프로젝트의 일관성을 확보하고 추후 분석 파이프라인에서 누락(Dropping)되거나 변환 에러가 생기지 않도록, 파싱 이후 `interim` -> `processed` 이동 구간에서 다음과 같은 4대 처리 수칙을 지킨다.

## 1. 결측치(NaN, Null) 처리 규칙
- 응답 JSON에서 특정 필드 누락(예: `recentRating` 등이 아예 없는 경우) 시 무조건 `interim` 단계에서는 `NaN` 으로 마킹한다.
- `processed` 적용 시:
  - 수치형 데이터(상금, 나이, 횟수)의 결측치는 `0`으로 치환. (등수/기록은 치환 금지)
  - 텍스트/범주형 데이터(마체중 불명 등)는 `"Unknown"` 으로 치환.
  - 마번(`hrNo`)이 결측인 행은 로우 레벨 분석 자체가 불가능하므로 예외 없이 **완전 삭제(Drop)** 한다.

## 2. 자료형 캐스팅 변환
- `totalPrize`, `age` 등 명백한 연속/이산 숫자로 계산될 항목은 Pandas 의 `pd.to_numeric(errors='coerce')` 로 밀어 넣고, 오류가 발생한 변환 실패 문자열(예: '비공개') 등은 결측으로 바꾼 뒤 상위 결측치 룰을 따른다.
- **날짜형(Date) 변환**: 
  - `debutDt` 나 `recentRcDate` 가 "YYYYMMDD" 형식 문자열로 올 경우 `pd.to_datetime` 로 변환을 적용한다. 만약 날짜로서 성립할 수 없는 이상한 값(예: "00000000")이 올 경우 강제 결측(NaT) 처리한다.

## 3. 컬럼 일원화 포매팅
- 카멜케이스(CamelCase)로 내려오는 API 원본 키 값을 반드시 스네이크 케이스(Snake_case) 형태로 파렌테시스 정규화한다. (예: `recentRcNo` -> `recent_rc_no`)

## 4. API227 데이터와의 조인 연계 규약 모색
- API227의 기초 테이블이 수집되었다는 전제하에, API227의 `meet`(경마장) 과 해당 경주에 참여한 말의 `hrName`(마명) 혹은 `hrNo`(마번)이 이번 데이터와 가장 유효한 연결고리이다.
- 이번 데이터는 **실시간 누적정보**이므로 API227을 기준으로 말의 과거 기록(`total1stCnt` 등)을 Left Join하게 되면, 시점에 따라 '성적 참조의 미세한 왜곡(Look-ahead bias)'이 있을 수 있다. 이 때문에 문서화 과정에서 **"해당 지표 정보는 2026년 이후 스냅샷 기준의 누적 정보임"**을 반드시 명시해야만 한다.
