## 2. 데이터셋 간 비교 분석

각 데이터셋의 구조를 비교한 결과, 데이터를 그룹화할 수 있습니다.

- **기수 성적 그룹** (`2_기수성적정보_Jeju.csv`, `3_기수성적정보_Busan.csv`, `3_기수성적정보_Seoul.csv`):

  - 세 파일 모두 동일한 컬럼 구조(13열) 및 동일한 행 수(629행)를 가집니다.

  - `jkNo`(기수번호)가 핵심 키로 사용 가능합니다.

- **경주 상세 성적표 그룹** (`7_경주별상세성적표.csv`, `8_경주별상세성적표.csv`, `9_경주별상세성적표.csv`):

  - 이름은 다르지만 시계열이나 지역별로 분할된 같은 형식의 데이터일 가능성이 높습니다.

  - 컬럼 구조 매핑을 통한 수직 병합(Union/Concat) 대상입니다.

- **경주 성적/기록 그룹** (`1_경주성적정보(15063979).csv`, `4_경주기록정보.csv`, `5_경주성적정보\`.csv`, `6_경주성적정보.csv`):

  - 파일 크기와 컬럼 수가 다양합니다. 1번 파일은 108개 컬럼으로 메인 데이터셋 성격입니다.

  - `entry_id`, `race_id`, `rcDate`, `jkNo`, `trNo` 등이 조인(Join) 키로 활용될 수 있습니다.


### 컬럼 형식 차이 및 정규화 필요 사항

- **혼합 타입(Mixed Types)**: `analyze_csvs.py` 실행 시 `jkNo`(기수번호), `trNo`(조교사번호) 등 식별자 컬럼에서 숫자와 문자가 혼용된 경고(DtypeWarning)가 발생했습니다. 이는 선행 문자열/숫자형식 통일이 필수적임을 의미합니다.

- **날짜 형식**: `rcDate`는 `int64`(예: 20260103)로, `rcDate_dt`는 `str`로 저장되어 있어, 일관된 `datetime` 형식으로 표준화가 필요합니다.

- **중복 데이터의 의미**: 기수성적정보 3개 파일(Jeju, Busan, Seoul)의 행 수와 컬럼이 정확히 동일합니다. 실제 데이터가 중복 수집된 것인지, 구조만 같고 값이 다른지 검증이 필요합니다.


## 3. 병합 가능성 판단

- **판단**: **조건부 병합 가능**

- **근거**:

  1. **공통 고유식별자 존재**: `jkNo`(기수번호), `trNo`(조교사번호), `race_id`(경주ID) 등 데이터셋을 묶어줄 연결고리가 존재합니다.

  2. **수직 병합 가능성**: 동종 데이터셋(기수성적, 경주별상세)은 수직 병합(Concat)이 가능해보입니다.

  3. **제약사항**: 식별자 컬럼의 데이터 타입 불일치(문자/숫자 혼용)와 7~9번, 1/4/5/6번 파일들 간의 관계 정의가 불명확하여, 단순 자동 병합은 결론을 왜곡할 위험이 매우 높습니다.


## 4. 예상 리스크

- **데이터 유실 (누락위험)**: `jkNo`, `trNo` 등의 타입 불일치로 인한 Join 실패.

- **다대다(M:N) 증식 위험**: 1번 파일(상세성적)과 4,6번 파일 간 병합 시 `race_id` 기준인지 `entry_id` 기준인지 정확히 매핑하지 않으면 데이터가 기하급수적으로 늘어날 수 있습니다.

- **중복 수집 데이터**: 기수 파일(2,3,3)과 성적 파일(7,8,9)이 만약 완전히 동일한 기간/내용을 덮어쓴 데이터라면 무작정 병합 시 중복 카운트 리스크가 큽니다.


## 5. 권장 병합 계획

### 1) 병합 목표

- 중복 행을 제거한 마스터 데이터셋 구축 (경주 메인 + 기수/조교사 특성 결합).


### 2) 사전 정제 항목 (필수 선행)

- **컬럼명 표준화**: 영문 소문자 형태 뱀표기법(snake_case) 또는 기존 `jkNo` 등의 camelCase 중 하나로 통일.

- **식별자 타입 통일**: `jkNo`, `trNo`, `owNo` 컬럼을 모두 문자열(String)로 변환하고, 앞뒤 공백 및 리딩제로(0) 제거 또는 보존 방식 통일.

- **날짜 형식 통일**: `YYYY-MM-DD` 형태의 문자열 또는 Date 객체로 변환.


### 3) 병합 기준안

- **1순위 기준키**: `entry_id` (개별 말의 경주 출전 단위 고유 식별자)

- **2순위 기준키**: `race_id` + `jkNo` (경주와 기수 매핑 시)

- **3순위 기준키**: `jkNo` (기수 정보 참조 시 Left Join)


### 4) 중복 / 충돌값 처리 방안

- **중복 행 제거**: 7,8,9번 등 파일 간 Concat 이후 `.drop_duplicates(subset=[식별자])` 적용.

- **충돌값**: 나중에 수집된 데이터(최신 기간)를 기준으로 덮어쓰기(Update) 적용.


### 5) 자동 병합/수동 검토 대상

- **자동 병합**: 기수 성적 정보(2,3번 파일)와 메인 경주 성적(1번 파일)은 `jkNo` 기준 자동 병합(Left Join) 권장.

- **수동 검토**: 4, 5, 6번 파일과 7, 8, 9번 파일은 병합 전에 샘플 데이터를 눈으로 대조하여 상호 배타적인지 포함관계인지 먼저 검증해야 함.


### 6) 병합 후 검증 방법

- 병합 전 기준 테이블(1번 파일)의 행 수와 병합 후 행 수 일치 여부 비교 (증식 여부 검증).

- `jkNo`, `trNo` 등의 매칭률(Inner Join 수량 / Left Join 수량) 계산하여 미매칭 건수 리포트.


## 6. 승인 요청 사항 (Final Request)

- **권고 판단**: 조건부 병합 가능

- **병합 가능 수준**: 중간 (사전 정제 전 단순 병합 불가. 정제 후 병합 가능)

- **선행 정제가 필요한 핵심 이슈**: `jkNo`, `trNo` 식별자 컬럼의 타입 불일치 (문자/숫자 혼용 에러) 해결 및 2~3, 7~9번 파일 간의 내용 완전 중복 여부 확인

- **제안 병합 방식**:
  1. 동종 데이터 수직 병합(Concat) 후 중복 제거
  2. 식별자 타입 문자열로 일괄 형변환
  3. 경주성적(Main) 데이터를 Left 로 놓고 기수정보 분기 병합(Merge)

- **내 승인 후 다음 단계에서 수행할 작업**: 위 계획에 동의하시면 파일별 '식별자 정제 및 수직 병합 스크립트'를 우선 작성하고, 매칭률 테스트 결과를 다시 보고드리겠습니다.
